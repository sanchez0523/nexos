# Nexos Development Roadmap

## Overview

| Phase | Scope | Milestone |
|-------|-------|-----------|
| 1 | Infrastructure skeleton | `docker compose up` 전체 기동 |
| 2 | Ingestion core | ESP32 데이터 → DB 저장 |
| 3 | API layer | REST + WebSocket 동작 |
| 4 | Alert engine | 임계값 초과 → Webhook 발송 |
| 5 | Dashboard | 실시간 차트 + 드래그앤드롭 |
| 6 | DX & Examples | 10분 퀵스타트 동작 |
| 7 | CI/CD & Release | GitHub Actions + v1.0.0 태그 |
| 8 | Marketing & Growth | GitHub Pages 랜딩 + 데모 GIF + HN/Reddit 런칭 |

**개발 규칙:**
- 각 Phase는 이전 Phase의 Milestone이 완료된 후 시작
- Phase 내 태스크는 병렬 가능한 것만 병렬 진행
- 각 태스크의 완료 기준(DoD)을 반드시 충족해야 다음 진행

## Phase 1 — Infrastructure Skeleton

**Milestone:** 모든 설정 파일 존재 + `docker compose config` 문법 검증 통과. 실제 `docker compose up`은 Phase 2(ingestion Dockerfile) + Phase 5(dashboard Dockerfile) + Phase 6(setup.sh로 인증서/passwd 생성) 완료 후에 가능.

### 1-1. docker-compose.yml
- `broker`, `ingestion`, `db`, `dashboard`, `proxy` 5개 서비스 정의
- 서비스 간 내부 네트워크 (`nexos-net`) 격리
- `db`에 named volume (`nexos-pgdata`) 마운트
- 모든 서비스에 `healthcheck` 정의
- `ingestion`은 `db` healthy 후 기동 (`depends_on condition: service_healthy`)
- **DoD:** `docker compose up -d && docker compose ps` 에서 모든 서비스 `healthy`

### 1-2. Mosquitto 설정
- `broker/config/mosquitto.conf` — TLS 전용, plaintext 포트(1883) 비활성화
- `listener 8883` + `certfile/keyfile/cafile` 경로 설정
- `allow_anonymous false` + `password_file` 경로 설정
- ACL: `ingestion-worker`는 `devices/#` subscribe 권한만
- **DoD:** `mosquitto_sub` TLS 연결 성공, 인증 없는 연결 거부 확인

### 1-3. TimescaleDB 초기화
- `db/init.sql` — `metrics`, `alert_rules`, `devices` 테이블 생성
- `create_hypertable('metrics', 'time')` 실행
- `metrics(device_id, sensor, time DESC)` 복합 인덱스
- retention policy: `add_retention_policy('metrics', INTERVAL '90 days')` (환경변수로 override 가능하게)
- **DoD:** `psql` 접속 후 `\d metrics` 에서 hypertable 확인

### 1-4. Caddy 설정
- `proxy/Caddyfile` — localhost HTTPS, SvelteKit(3000) + Go API(8080) 리버스 프록시
- `/api/*`, `/ws` → ingestion, 나머지 → dashboard
- **DoD:** `https://localhost` 브라우저 접근 시 인증서 경고 없이 응답

### 1-5. .env.example 완성
- 모든 환경변수 주석 포함
- 생성 필요 값은 `# generated by setup.sh` 명시
- **DoD:** `.env.example` 복사 후 `setup.sh` 없이 수동으로도 설정 가능한 수준

## Phase 2 — Ingestion Core (Go)

**Milestone:** ESP32가 `devices/esp32-01/temperature`로 publish → `metrics` 테이블에 row 삽입 확인

### 2-1. Go 프로젝트 초기화
- `ingestion/go.mod` — module명 `github.com/yourname/nexos/ingestion`
- 의존성: `gofiber/fiber/v2`, `paho.mqtt.golang`, `pgx/v5`, `golang-migrate`
- `cmd/server/main.go` — 서비스 wiring (MQTT subscriber goroutine, Fiber server)
- **DoD:** `go build ./cmd/server/...` 성공

### 2-2. DB 연결 + Migrations
- `internal/db/db.go` — `pgx/v5` pool 설정, `ping` 헬스체크
- `migrations/000001_init.up.sql` — Phase 1-3 스키마를 migration 파일로 이관
- `migrations/000001_init.down.sql` — 롤백
- 서버 기동 시 자동 migrate up
- **DoD:** `make migrate-up` 성공, `go test ./internal/db/...` 통과

### 2-3. MQTT Subscriber
- `internal/mqtt/subscriber.go` — `paho` TLS 연결, `devices/#` subscribe
- 재연결 로직: 지수 백오프, 최대 30초, 무한 재시도
- 수신 메시지를 `chan Metric` 으로 fan-out
- **DoD:** 실제 Mosquitto 연결 후 메시지 수신 로그 확인 (integration test)

### 2-4. Topic Parser
- `internal/mqtt/topic.go` — `devices/{device_id}/{sensor}` 파싱
- 형식 불일치 토픽은 `ErrInvalidTopic` 반환 (drop, 로그)
- **DoD:** `go test ./internal/mqtt/...` — 정상/비정상 토픽 케이스 전부 커버

### 2-5. Payload Parser
- `internal/mqtt/payload.go` — `{"value": N}` 또는 bare number 파싱
- 파싱 실패 시 `ErrInvalidPayload` 반환 (drop, 로그)
- **DoD:** `go test ./internal/mqtt/...` — float, int, 문자열 혼입, 빈 값 케이스

### 2-6. Metrics Writer
- `internal/db/metrics.go` — `INSERT INTO metrics` (batch insert, 100ms 또는 50건마다 flush)
- `internal/db/devices.go` — `UPSERT INTO devices (last_seen)` on every message
- **DoD:** `go test ./internal/db/...` — 실제 TimescaleDB 연결 통합 테스트

## Phase 3 — API Layer (Go)

**Milestone:** `curl https://localhost/api/devices` JWT 인증 후 디바이스 목록 반환, WebSocket 연결 후 실시간 메시지 수신

### 3-1. Fiber 서버 + 미들웨어
- `internal/api/server.go` — Fiber 앱 초기화, 미들웨어 체인
- 미들웨어: recover, logger (slog JSON), CORS (localhost only)
- `/health` — `{"status":"ok","db":"ok","mqtt":"ok"}`
- **DoD:** `curl http://localhost:8080/health` → 200

### 3-2. JWT 인증
- `internal/auth/jwt.go` — access(24h) + refresh(7d) 토큰 발급/검증
- `POST /api/auth/login` — username/password → access + refresh 토큰
- `POST /api/auth/refresh` — refresh token → 새 access token (rotation)
- `internal/api/middleware/auth.go` — Bearer 검증 미들웨어
- **DoD:** `go test ./internal/auth/...` — sign/verify/expiry/rotation 케이스

### 3-3. Devices API
- `GET /api/devices` — 전체 디바이스 목록 + `last_seen` + online/offline 상태
  - online 기준: `last_seen` 60초 이내
- `GET /api/devices/:id/sensors` — 디바이스의 센서 목록
- **DoD:** `go test ./internal/api/...` — `httptest` 기반 엔드포인트 테스트

### 3-4. Metrics API
- `GET /api/metrics` — query params: `device_id`, `sensor`, `from`, `to`, `limit`
- TimescaleDB `time_bucket` 활용한 다운샘플링 (조회 범위 > 1시간이면 1분 버킷)
- **DoD:** 1000개 이상 row 쿼리 시 500ms 이내 응답

### 3-5. WebSocket Hub
- `internal/api/ws/hub.go` — 클라이언트 등록/해제, broadcast goroutine
- `GET /ws` — JWT 쿼리파라미터 인증 후 업그레이드
- MQTT subscriber → hub channel → 모든 연결 클라이언트에 broadcast
- 메시지 포맷: `{"device_id":"esp32-01","sensor":"temperature","value":23.5,"time":"..."}`
- **DoD:** `websocat wss://localhost/ws?token=...` 연결 후 실시간 메시지 수신 확인

## Phase 4 — Alert Engine (Go)

**Milestone:** 온도 임계값 설정 → 초과 시 webhook URL로 Generic JSON POST 전송

### 4-1. Alert Rules CRUD
- `GET /api/alerts` — 전체 alert rules
- `POST /api/alerts` — 생성 (device_id, sensor, threshold, condition, webhook_url)
- `PUT /api/alerts/:id` — 수정
- `DELETE /api/alerts/:id` — 삭제
- **DoD:** `go test ./internal/api/...` — CRUD 엔드포인트 테스트

### 4-2. Threshold Engine
- `internal/alert/engine.go` — 메트릭 수신 시 해당 device_id+sensor의 rules 평가
- `above` / `below` 조건 평가
- `ALERT_TIMEOUT_SECONDS` 쿨다운 (동일 rule 중복 발송 방지, in-memory TTL map)
- **DoD:** `go test ./internal/alert/...` — above/below/cooldown/edge case 전부

### 4-3. Webhook Dispatcher
- `internal/alert/webhook.go` — HTTP POST, 타임아웃 10초, 실패 시 1회 재시도
- Generic JSON 포맷: `{"device_id","sensor","value","threshold","condition","triggered_at"}`
- 발송 결과 slog 기록
- **DoD:** `httptest` mock 서버로 발송 페이로드 검증

## Phase 5 — Dashboard (SvelteKit)

**Milestone:** 브라우저에서 로그인 → 디바이스 카드 자동 생성 → 실시간 차트 갱신 → 드래그앤드롭 레이아웃 저장 → 알림 규칙 설정

### 5-1. 프로젝트 초기화
- SvelteKit + TypeScript + Tailwind CSS 설정
- `src/lib/api/` — REST client (typed, fetch wrapper)
- `src/lib/stores/` — `authStore`, `deviceStore`, `wsStore`, `alertStore`
- **DoD:** `npm run build` 성공, `npx svelte-check` 0 errors

### 5-2. 인증 페이지
- `/login` — username/password form → JWT 저장 (httpOnly cookie 또는 memory)
- 인증 만료 시 자동 refresh, 실패 시 `/login` 리다이렉트
- **DoD:** 로그인 → 대시보드 진입, 만료 토큰으로 API 호출 시 자동 갱신

### 5-3. WebSocket Store
- `wsStore` — 연결, 메시지 수신, 지수 백오프 재연결 (최대 30초)
- 수신 메시지 → `deviceStore` 자동 업데이트
- **DoD:** 백엔드 재시작 시 자동 재연결 확인

### 5-4. 차트 컴포넌트
- `LineChart.svelte` — Chart.js, 실시간 dataset append (최대 200포인트 유지)
- `Gauge.svelte` — Chart.js doughnut 기반, min/max/current 표시
- `StatusIndicator.svelte` — online(초록)/offline(빨강) 배지
- **DoD:** 각 컴포넌트 Storybook 없이 `/test` 라우트에서 mock 데이터로 시각 확인

### 5-5. Auto-Discovery + 카드 레이아웃
- 신규 `device_id` + `sensor` 감지 시 gridstack에 카드 자동 추가
- `gridstack.js` 통합 — 드래그앤드롭, 크기 조정
- 레이아웃 localStorage 저장/복원
- **DoD:** 새 토픽 publish → 새 카드 자동 생성 (페이지 새로고침 없이)

### 5-6. 과거 데이터 조회
- 차트 카드 내 시간 범위 선택기 (1h / 6h / 24h / 7d / custom)
- REST API 호출 → 차트 교체
- **DoD:** 7일치 데이터 조회 시 차트 정상 렌더링, 빈 범위 처리

### 5-7. 알림 규칙 UI
- 카드별 알림 설정 모달 (threshold, condition, webhook_url, enable/disable)
- 전체 알림 목록 페이지 `/alerts`
- **DoD:** 알림 생성 → 저장 → 목록 표시 → 삭제 흐름 완성

## Phase 6 — DX & Examples

**Milestone:** 클론 → `./scripts/setup.sh` → `docker compose up -d` → ESP32 연결 → 차트 확인까지 10분 이내

### 6-1. setup.sh (인터랙티브)
- admin username/password 입력
- JWT_SECRET 32바이트 자동 생성
- self-signed TLS 인증서 생성 (openssl, 365일)
- Mosquitto passwd 파일 생성 (ingestion-worker 계정 포함)
- `.env` 파일 생성
- **DoD:** 깨끗한 환경에서 스크립트 실행 후 `docker compose up -d` 정상 기동

### 6-2. add-device.sh
- device ID, password 입력
- Mosquitto passwd 파일에 계정 추가 (`mosquitto_passwd`)
- 실행 중인 broker에 reload 신호 전송
- **DoD:** 스크립트 실행 후 재시작 없이 신규 디바이스 연결 성공

### 6-3. examples/esp32
- `examples/esp32/nexos_example.ino`
- WiFi + MQTT TLS 연결 (PubSubClient)
- `devices/{DEVICE_ID}/temperature`, `humidity` 주기 publish (5초 간격)
- 설정값 상단 `#define` 블록으로 집중
- **DoD:** 스케치 업로드 → 대시보드 카드 자동 생성 확인

### 6-4. examples/raspberry-pi
- `examples/raspberry-pi/nexos_client.py`
- `paho-mqtt` TLS 연결
- CPU 온도, 메모리 사용률 publish
- `requirements.txt` 포함
- **DoD:** `python3 nexos_client.py` 실행 → 대시보드 카드 자동 생성 확인

### 6-5. examples/simulator
- `examples/simulator/main.go`
- 복수 가상 디바이스 (기본 3개), 랜덤 노이즈 포함 sine wave 데이터 생성
- CLI 플래그: `--devices`, `--interval`, `--broker`
- **DoD:** `go run main.go` → 대시보드에 3개 디바이스 자동 등록 확인
